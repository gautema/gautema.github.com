Testing

En bedre måte?

Gaute Magnussen
02. 02. 2012

Bakgrunn
Har skrevet mange tester. Mange dårlige. Mange gode.
Kjent både fordeler og ulemper på kroppen.
Min måte
BDD-style, Akseptansetest først, Unit test det vanskelige, Integrasjonstester på resten, Mock eksterne avhengigheter, Abstrahere vekk infrastruktur.
behaviour driven development
scenario, given when then, features,
dekopling fra implementasjon, outside in, gherkin,
business value.
Akseptansetest
først
test utenfra, implementasjon er uviktig, forventet resultat er viktig, Et scenario i hver klasse, mange tester, mange rammeverk
kan kjøre i minnet, test ende til ende.
Unit test det vanskelige
Modeller, servicer, hjelpemetoder, legg den vanskelige koden i klasser som er enkle å teste og har få eksterne avhengigheter.
Mocking
lite mocking, mock ustabile ting stabilt, mock eksterne avhengigheter, verifikasjoner gir skjøre tester, ikke vær redd for å bruke eksisterende implementasjoner, mocks er implementasjoner som må vedlikeholdes.
Abstrahere vekk infrastruktur.
orm muliggjør sqlite og sql server ce, ravendb in memory, container kan benytte mocker i test, openrasta og nancyfx kan kjøre in memory. Tenk på dette når man velger infrastruktur.
Utfordringer:
    Akspetansetester er trege - trege å debugge, trege å kjøre. Tungt å starte opp alle rammeverk i hver test. Ingen absoulutt regel på hva som skal skrives når. Ingen automatikk i hvilke testerfiler som berører hvilke klasser.
Fordeler:
    Refactoring er enkelt. Dokumenterer systemet bedre. Røyker ut integrasjonsproblemer. Bedre flyt i testskrivingen. Fjerner behov for unødige abstraksjoner.
